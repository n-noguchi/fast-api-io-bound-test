# チューニング結果レポート (RESULTS.md)

FastAPI(API1) が Go 製 API2（0.5s sleep の IO バウンド）を呼び出す構成で、
**API1 の呼び出し方がスループット/レイテンシに与える影響** を k6 で計測・探索した結果。

- 計測ツール: k6 (Docker, `grafana/k6:0.54.0`)
- シナリオ: ramping-vus `0 -> 100 VU (5s) -> 100 VU (20s) -> 0 (5s)`
- API2 の sleep: **500ms / リクエスト**
- 理論上の上限: `100 VU / 0.5s = 200 req/s`, レイテンシ下限 `= 500ms`

---

## 1. API1 実装パターンの比較 (uvicorn workers=1)

同じ仕事（API2 を 1 回呼ぶ）を、HTTP クライアントの扱い方だけ変えて比較。

| endpoint | req/s | p50 (ms) | p90 (ms) | p95 (ms) | max (ms) | fail (%) |
|---|---:|---:|---:|---:|---:|---:|
| `/call/sync-requests`  (同期 requests)        |  73.8 | 1251.9 | 1288.2 | 1296.7 | 1327.9 | 0.0 |
| `/call/sync-httpx`     (同期 httpx)           |  72.7 | 1257.0 | 1320.9 | 1336.3 | 1402.9 | 0.0 |
| `/call/async-naive`    (非同期・都度クライアント) |  84.4 | 1108.7 | 1231.7 | 1243.4 | 1550.7 | 0.0 |
| `/call/async-pooled`   (非同期・共有クライアント) | 157.7 |  525.7 |  559.9 |  563.9 |  591.2 | 0.0 |
| `/call/async-tuned`    (非同期・共有＋プール拡張) | **164.8** | **504.0** | **506.5** | **508.0** | **530.5** | **0.0** |

### 読み取り方
- **同期系（requests / httpx）は約 73 req/s で頭打ち**: Starlette のスレッドプール（既定 40）で
  IO 待ちの間スレッドを占有するため、100 VU を捌ききれずレイテンシが 1.2〜1.3s に膨らむ。
- **async-naive は 84 req/s**: 非同期だがリクエストごとに `AsyncClient` を new するため
  毎回 TCP/HTTP の接続確立が走り、keep-alive の恩恵がない。
- **async-pooled で 158 req/s へ急増**: 起動時に作った `AsyncClient` を全リクエストで使い回し
  接続を再利用。これが **決定的な差**（同期の 2.1 倍）。
- **async-tuned で 165 req/s（最速）**: さらに `httpx.Limits(max_connections=1000,
  max_keepalive_connections=500)` でプールを広げ、p95 508ms と API2 の sleep(500ms)+8ms に収まる。

> **結論1: IO バウンドなら「非同期 ＋ クライアント使い回し(keep-alive) ＋ プール拡張」が圧勝。**
> workers=1 の単一イベントループで理論上限(200 req/s)の **約 82%** を達成。

## 2. uvicorn ワーカー数の探索（最速パターン `/call/async-tuned`）

ワーカー数を変えて最速パターンを再計測（warm-up 強化後、連続計測）。

| uvicorn workers | req/s | p50 (ms) | p95 (ms) | max (ms) | fail (%) |
|---:|---:|---:|---:|---:|---:|
| 1 | 165.1 | 503.2 | 505.7 | 521.7 | **0.0** |
| 2 | 176.1 | 503.2 | 505.9 | 521.3 | 6.4 |
| 4 | 192.5 | 503.1 | 505.7 | 534.6 | 14.5 |

### 読み取り方
- **workers を増やすと req/s は上がるが、失敗率が急増する。**
- API1 のアクセスログは全件 `200 OK`（エラー/5xx なし）。つまり失敗は **k6→API1 の接続確立時**
  に起きており、複数ワーカーが 1 つのリスニングソケットを共有するため **accept キュー(バックログ)
  が高同時で溢れた** こと（接続拒否）が原因と推定される。
- レイテンシ(p50/p95)はどのワーカー数でも約 503〜506ms でほぼ同じ。つまり CPU はボトルネック
  になっておらず、**ワーカーを増やしてもレイテンシは改善しない**。

> **結論2: IO バウンド処理では workers=1（非同期単一ループ）が最も安定。**
> workers 増はスループットの頭打ちをわずかに緩和するが、接続受け入れの失敗を招き信頼性を下げる。
> 信頼性を含めた総合最优は **workers=1**。

## 3. 最終的な最適構成（推奨）

```
API1 = FastAPI / async def エンドポイント
       + 起動時に生成した httpx.AsyncClient をアプリ全体で使い回し(lifespan)
       + httpx.Limits(max_connections=1000, max_keepalive_connections=500)
       + uvicorn workers=1
```

| 指標 | 値 |
|---|---|
| スループット | **165 req/s** (100 VU 時) |
| p95 レイテンシ | **506 ms** (API2 sleep 500ms + 6ms) |
| 失敗率 | **0.0 %** |
| 同期実装に対する高速化 | **約 2.2 倍** (73 -> 165 req/s) |

### 改善の主要因（寄与の大きい順）
1. **非同期化 (async/await)** — GIL 下でも単一スレッドで多数の IO 待機を並行（最大の寄与）。
2. **HTTP クライアントの使い回し (keep-alive)** — 接続確立コストを除去（async-naive→pooled で +73 req/s）。
3. **接続プール上限の拡張** — 高同時接続で keep-alive を維持（+7 req/s, p95 を 564→508ms に短縮）。
4. **workers=1** — IO バウンドでは CPU が余るため、プロセス増より単一ループが安定。

### 注意事項
- ワーカー数を増やす場合は API1 の **リスニングバックログ（uvicorn の `--backlog` /
  カーネル `somaxconn`）** と接続受け入れ能力を併せてチューニングしないと、接続拒否による
  失敗が増える（本検証の workers>=2 で再現）。厳密な限界値の特定は今後の課題。
- API1/API2 間は Docker 内部ネットワーク（`http://api2:8080`）で通信。本結果はコンテナ間通信の
  オーバーヘッドを含む実測値。
